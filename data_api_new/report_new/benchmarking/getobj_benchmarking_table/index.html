    Getting table data in a user report

    Getting table data in a user report > Getting table data in a user report
    Method getobj.benchmarking_table
    Description Getting table data in a user report
    Available to Agent, Client
    Necessary components for working with the method Data API for configuration
    Method weight 1 point

    Request parameters

    method is passed
    Name Type Required Valid values Description
    access_token string yes Authentication session key
    user_id number no Unique user identifier of the agent client on behalf of which the request is made
    Is required for the agent
    To obtain a list of client users, you must use the method "get.customer_users"
    limit number yes You cannot request more than 200 lines. Number of records to return. Default 50.
    offset number yes Shift - determines from which record number to return limit records. Default 0.
    report_type enum yes User report type. To get a list of user reports, you must use the "get.reports_list"
    date_from iso8601 yes YYYY-MM-DD hh:mm:ss Start date of main sample (limit 366 days)
    date_till iso8601 yes YYYY-MM-DD hh:mm:ss End date of the main sample
    perspective_window number depends on parameters You can set a period from 1 to 180 days from session to request The number of days from the session to the request in which the session request is attributed.
    Is required for `report_type` = `ad_analytics`
    dimensions array yesDo not specify if custom_dimension_groups is passed List of dimension mnemonics, separated by comma. For an available list of dimensions, see "Getting dimension nesting" . The order affects the construction of the tree.
    custom_dimension_groups array yes Do not specify if dimensionsGroups of conditions used in the custom dimension are checked. The structure and requirements are similar to groups "Creating a custom dimension" .
    columns array yes List of custom column identifiers, separated by comma. For the available list of columns, see "Getting Column Nesting" . The order affects the answer.
    Filters used in the report
    filter object no It is permissible to use no more than 2 уровней nesting. See examples below Filters applied in the report. The construction logic is similar to "Filtering criteria"
    filters array yes An expression can contain both simple filters and a filter tree.
    field string The field of the entity to which filtering will be applied (the list is predefined for the method
    operator enum Filtering operator. A list of all operators can be obtained in the section "Filtering operators"
    value string Value for the filtering operator. Optional field; if it is missing, it is considered empty.
    Filters applied to columns
    filters array yes An expression can contain both simple filters and a filter tree.
    field string The field of the entity to which filtering will be applied (the list is predefined for the method
    operator enum Filtering operator. A list of all operators can be obtained in the section "Filtering operators"
    value string Value for the filtering operator. Optional field; if it is missing, it is considered empty.
    Sorting used in the report
    sort array no The sorting used in the report. The construction logic is similar to "Data sorting"
    Row expansion
    parent_dimensions_value_ids array no ["N","M"] Select a report line for further expansion. Consists of a sequential list of dimension row identifiers. If not specified, level 1 is returned.

    Response parameters

    is returned
    Name Type Valid values Filtering Sorting Default response Description
    date_from iso8601 YYYY-MM-DD hh:mm:ss yes yes Start date of the main sample
    date_till iso8601 YYYY-MM-DD hh:mm:ss yes yes End date of the main sample
    Array of strings of the requested report
    rows array yes Массив строк запрашиваемого отчета
    List of requested dimensions
    dimension array yes List of values requested by dimensions
    value string yes yes If there is no data in the requested dimension, Не определеноName of values requested dimensions
    value_id string Unique identifier of the value branch of the requested dimensions
    List of requested columns
    columns array yes yes List of values of the requested columns in the dimension context
    expandable boolean Is it possible to expand a dimension
